    # 项目介绍

    # 背景

    中厚钢板大约有200的生产历史，它是国家现代化不可缺少的一项钢材品种，随着我国工业的发展，对中厚钢板产品，无论从数量上还是从品种质量上都已提出了更高的要求；但是厚板生产过程中板形质量总是出现异常，而且难以定位到生产过程导致异常的根本原因，因此基于此问题我们设计开发了，厚板生产质量的监控，异常诊断以及质量回溯系统

    # 面临的挑战：

    **挑战一：**针对轧钢行业生产过程数据，并且了解轧钢的流程都大致了解轧钢是一个传统的流程行业，针对这种具有小批次，多规格，以及过程变量繁多，工艺复杂，而且数据具有多源异构等特点的加工过程，我们是无法通过传统的根据数据建模的方法，来分析导致钢板最终出现异常的原因；

    因此我们设计并开发了一个overview-details（宏观到微观）的分析思路，我们通过一个从大量的过程数据到一个具体工艺的数据来进行聚焦分析来进行对钢板产生异常的原因来进行特定的分析；

    1. 第一步通过交互进行获取一个近一个月生产数据，通过特定的聚类算法（基于密度峰值聚类的算法，DPCA）来进行对生产的钢板进行聚类，

    2. 对于聚类好的数据，这么多的过程指标我们又是如何在一个二维平面内进行展示的呢；我们这里就利用了一个TSNE进行把117维过程数据进行降维处理，通过聚类的钢板数据可以清晰看到哪一个种类的质量坏的多，我们知道了当前种类钢板的坏板较多，这部分主要是监控的功能

    3. 那么究竟是什么导致产生此原因的呢，加热过程加热不均匀，还是轧制过程轧制的道次分配，或者是轧制过程的人工调整量，以及冷却过程的冷却不均导致的呢？面对如果多的过程变量（在我们统计分析中，选取的是117维度过程变量），如何准确确定某个过程变量导致的原因最为显著，因此我们通过一个基于pca的故障诊断算法并同时与专家经验相融合，我们可以的到聚焦到某一个或者某些导致钢板异常的原因，

    4. 我们系统的监控和诊断部分，监控主要是监控生产的状态和质量的指标；而对于诊断功能，到底具体什么原因导致的这样的事情产生的呢；而对于回溯就是为专家进行推荐一个科学有效的生产顺序，以及生产工艺参数从而实现一个辅助决策。

      就像看病一样，我们首先需要知道了一个病人有病，并通过设备检查检查知道了什么原因导致的病因，最后我们才能够对症下药了；

    # 算法简介

    # 基于密度峰值聚类

    我们知道传统的k-means聚类算法的前提是你知道k值，但是钢板数据，我们根本不知道在大量的数据中应该聚几类，什么样的规格算一类，而基于密度峰值的聚类算法就是，在前提为**（1）聚类中心的密度应该比较大（2）•聚类中心应当离比其他密度更大的点较远**

    1. 计算全局任意两点之间的距离
    2. 确定距离阈值 （dc,使得平均每个点的邻居数为所有点的1%~2%。参数dc的选取）决定了算法的成败，太大的话就一个聚类中心，太小的话同一类可能导致很多种聚类中心；
    3. 每个点的局部密度ρi
    4. 每个点i计算在局部密度高于对象i的所有对象中，到对象i最近的距离δ
    5. 绘制出局部密度ρi与高局部密度点距离δi的关系散点图。并获得离群点就是聚类中心

    # pca故障诊断

    pca降维就是把将n维特征映射到k维空间中，同时数据空间分解为主元子空间（k坐标系）和残差子空间（r坐标系数），每一组观测数据都可以投影到这两个子空间内，因此引入T^2和平方预报误差Spe，两个统计量来检测故障；

    pca过程 (opens new window)

    1. 标准化，即每一维特征减去各自的平均值，再除以每一维的标准差；
    2. 计算协方差矩阵
    3. 利用特征值分解方法求写访哈矩阵的特征值与特征向量
    4. 对特征值从大到小排序，选择其中最大的k个，将其对应的k个特征向量分别组成降维矩阵P
    5. 将数据转换到k个特征向量构建的低维子空间中，即Yt = PtXt
    6. n=k+r；主元分析法将数据空间分解为主元子空间（k坐标系）和残差子空间（r坐标系数），每一组观测数据都可以投影到这两个子空间内，因此引入T^2和平方预报误差Spe，两个统计量来检测故障；
    7. T^2^和SPE都超限；或者T^2^正常SPE超限都认为是异常的

    # 项目开发遇到的问题

    1. 工艺方面。主要是轧钢行业是经典的流程工业，工艺复杂，过程变量繁多，而且过程变量也是多种来源而且有高维度和低纬度的方方面面的数据，了解工艺过程任务艰巨，就从现场考过来的数据库中的表。我们对表中的每一个字段都进行了和现场校正，最终得到的一个word文档都上百页面

    2. 开发过程：由于前端展示的数据巨大页面很卡。而且利用d3进行绘图，虽然图元具有高定制化，但是不得不面对性能的一些问题，就比如一个马雷图（代表钢板的生产时间线）我们需要展示上千块的钢板在页面上，如果遇到交互的时候就会页面特别卡顿，这里使用了一个虚拟列表的方法，让不在可视区域中的图形进行渲染从而进行了优化；

      其次就是由于我们需要对不同的钢板进行诊断分析的时候就会面临这数据更新了，但是前端的页面并没有更新，这里我们使用了VUe.nextTick进行在下次Dom更新循环之后进行回到，从而获取更新后的dom

      在methods中定义的数据不能动态的绑定，使用Vue原生的Vue.set()方法进行把属性添加到动态相应

      有些数据不在需要进行vue的动态绑定了使用Object.freeze进行对象冻结；

      而且后来面临着新的需求的提出，需要对系统进行迭代，使用由于使用d3的灵活性能，就面临着工程的话的方面，我们对于一些基础的图元的抽象和封装，以及一些简单的交互的抽象和封装，让代码的可读性了，可更改性能大大提高

      由于交互逻辑的复杂以及数据量太大，进行点击事件的节流处理（就是实现了进行在一定的时间内点击多次但是知识相应一次的事件，而且响应的是第一次点击的那次）；

      css3硬件加速渲染（transform:transition3d(0,0,0);）

    3. 部署问题：进行wepack进行打包的时候我们配置了vue路由的懒加载（当进行打包构建应用时，打包后的代码逻辑实现包可能会非常大。当我们把不同的路由对应的组件分别打包，在路由被访问时再进行加载，就会更加高效。）以及一些第三方的插件的懒加载；
